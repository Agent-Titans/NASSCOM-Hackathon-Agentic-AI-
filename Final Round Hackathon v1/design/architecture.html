Artificial intelligence — IT ticket routing

Proposed solution architecture. One system, one ticket, one decision: which Hand takes it from here.

Companion docs: low_level_design(LLD).html · Open source tools and libraries.html

What we are building

We route and resolve IT tickets the moment they arrive — fast when we are sure, assisted when we are partly sure, human when we are not.

Less waiting. Less rework. No false confidence.

Architecture flow

Five AI agents

Agents 1–4 always run in order; agent 5 (Supervisor) picks the Hand once their outputs exist.

Step 1

Guardrail AI agent

Privacy & safety first

Scans the raw ticket for personal data and secrets, masks them, and passes only safe text downstream and into logs.

Step 2

Classifier AI agent

What kind of issue

Picks one primary issue type (plus a short sub-label) so routing and search stay consistent.

Step 3

Router AI agent

Where it goes

Maps category + urgency to department, priority, and SLA hours using configured routing rules.

Step 4

Resolver AI agent

Memory + AI

The backend turns safe text into a vector, searches ChromaDB, then calls Gemini with ticket + matches—the UI only surfaces the finished steps.

Step 5

Supervisor AI agent

Supervisor AI agent

Runs after the other four. It reads their outputs, builds one confidence number c_total (0–1), picks Hand 1, 2, or 3, applies safety rules, and tells the system what to save and notify. It does not replace them—it decides the outcome once the pipeline has spoken.

Classifier output domains (Use Case 1): Infrastructure · Application · Security · Database · Storage · Network · Access Management.

Notifications (email / portal) run after the Supervisor AI agent—they carry out its decision; they are not a sixth reasoning AI agent.

The Three Hands Solution

These are the only outcomes visible in the product. Everything else in the pipeline exists to pick exactly one of them.

Hand 1

Self-help

High trust

Numbered steps from the knowledge base and similar resolved tickets. The requester follows them in the app or email link, then taps Worked to close or Did not work to route the same ticket onward—no retyping.

Hand 2

Automatic routing

Medium trust

Ticket lands in the right team queue with priority and SLA. Assignee sees suggested steps and similar past tickets. Requester sees who owns it and expected response time.

Hand 3

Human escalation

Low trust or policy

Specialist queue with escalation flag and audit context. No fake certainty when scores are low, search is weak, or a rule requires a person. The requester sees a straight status until someone resolves it.

How we decide — the Supervisor AI agent

The Supervisor runs after Guardrail → Classifier → Router → Resolver. It reads their outputs and picks exactly one Hand, what to persist, and what notifications send.

c_total (0–1) blends retrieval strength, optional tone, and past feedback for similar issues:

  • ≥ 0.80 → Hand 1 (self-help)
  • 0.60–0.79 → Hand 2 (routed assist)
  • < 0.60 or policy hit → Hand 3 (human)

Security-sensitive categories and weak RAG grounding force Hand 3 regardless of score. Formulas and overrides are in the low-level design.

How retrieval + Gemini fit (RAG)

The Resolver AI agent never runs in the browser. Only the backend Resolver AI agent calls Gemini, and only after the Guardrail and Classifier AI agents have finished.

Order of operations:

  1. The server turns the safe ticket text into an embedding (a numeric fingerprint of “meaning”).
  2. ChromaDB returns the closest past tickets and help snippets.
  3. The server builds a single prompt: ticket + those snippets + configured policy text.
  4. Gemini returns draft steps; the app stores them and shows them in the UI.

Memory first (ChromaDB), generation second (Gemini) through the Resolver AI agent. If search is empty or weak, the Supervisor AI agent steers toward Hand 2 or Hand 3 instead of inventing steps.

Examples — same ticket shape, different Hands

Reference scenarios; thresholds are configuration-driven.

Example · Hand 1

“Locked out — need password reset”

Strong match to past resets

The Classifier and retrieval signals agree on a known pattern. The Supervisor AI agent assigns Hand 1: the requester sees a short checklist (portal → forgot password → email link). If it fails, one control escalates the same ticket to the access team.

Example · Hand 2

“Office printer stops mid-job”

Clear area, not a one-tap self-close

The Classifier AI agent routes to Infrastructure; the Resolver AI agent pulls similar printer tickets and draft steps. The Supervisor AI agent assigns Hand 2: the ticket sits in the Hardware queue with SLA, the assignee sees suggestions and references, the requester sees routing to the hardware team.

Example · Hand 3

“Possible data breach — unusual login activity”

Security policy or weak grounding — human first

The Classifier AI agent maps to Security; retrieval is thin or policy forces a person. The Supervisor AI agent assigns Hand 3: the requester sees escalation status only—no automated self-help steps. The ticket lands in the SecOps specialist queue with escalation flag, SLA, and full audit context (sanitized ticket, agent trace, similarity score) for the assignee.

Feedback loop

Hand 1 (self-help): after the requester tries the steps, the ticket detail offers Worked or Did not work (optional note). That outcome is saved on the ticket and feeds the next similar ticket’s internal scoring—so repeat failures do not keep getting the same Hand 1 path.

Hand 2 / Hand 3 (same idea, different surface): we surface closure confirmation (“Was this resolved satisfactorily?”) or assignee-entered resolution + outcome; the same storage feeds historical success for the Supervisor AI agent. We ship Hand 1 feedback first; the pattern extends to Hand 2 and Hand 3 without a new engine.

Logs: every AI agent step, Hand change, and feedback event is written to append-only audit logs (who/what/when + compact JSON). That supports troubleshooting, compliance, and tuning—separate from the ticket timeline in the product.

Supporting layers (infrastructure)

Placed here because they sit under the pipeline and UI above: they are not “another product screen,” but the services that make the architecture safe, durable, and operable.

Identity & access — Who may create tickets, who may see queues, and session handling (e.g. JWT, HTTP-only cookies, rate limits on login). Stops cross-tenant reads and supports audit “who did this.”

Persistence — Single source of truth in SQLite (via SQLAlchemy): tickets, classifications, routings, resolutions, feedback, and audit rows. Every Hand transition and SLA clock lives here so the UI is always reading the same state.

Vector memory — ChromaDB stores embeddings and similar-ticket retrieval context for the Resolver AI agent. It supports RAG and does not replace SQLite as the transactional system of record.

Operational interfaces — Beyond Streamlit: email/notifications, future webhooks, exports. Anything that reacts to a stored state change (new ticket, Hand change, SLA breach) without duplicating business rules in the channel.

Demonstration data — where tickets come from

Two data paths: tickets the application creates and routes during a walkthrough and historical text the RAG layer retrieves.

Live tickets (primary demonstration path): we create tickets through the normal New ticket form. They persist in SQLite and appear under My tickets or My queue.

Corpus for ChromaDB and “similar tickets”: we will load a controlled synthetic CSV or JSON aligned to our Hand 1/2/3 scenarios through a one-off ingest script. We will optionally import a public, anonymized IT-helpdesk-style dataset (for example from Kaggle or other open CSVs), map columns, run the same Guardrail AI agent rules on ingest, and embed rows into ChromaDB so the Resolver AI agent has retrieval depth. We will not mix production employee or customer data into demonstration environments.

Provenance: we will keep synthetic seeds as the default so demonstration paths stay predictable. When we add a broader import, we will document source, filters, and which records are scripted versus imported.

Time resolution — where we save time

What we save: queue wait for obvious repeats (Hand 1), and triage minutes for IT when the ticket arrives with team, priority, draft steps, and similar cases already attached (Hand 2).

How: automation only runs when the Supervisor is comfortable; otherwise time is “spent” on the right human instead of on wrong automation rework.

Where it is recorded: timestamps on the ticket row and in audit logs—created, first response, routed, resolved—plus SLA targets from the Router. Reporting can show time-to-first-meaningful-response and time in queue per Hand without new sensors.

Responsible & ethical AI

  • Guardrail AI agent first — no raw PII in prompts or careless logs.
  • Grounded answers — RAG-first Resolver AI agent.
  • Graceful degradation — weak retrieval or low score → Hand 3 from the Supervisor AI agent, not a wrong confident answer.
  • Auditability — every AI agent decision traceable.

Evaluation intent

Round 2 design targets—measured against a golden ticket set during build and demo.

Metric How we measure (design) Round 2 target
Classification accuracy Golden set of 15 tickets mapped to Use Case 1 domains; compare Classifier output to expected category ≥ 85%
Routing correctness Expected department queue + priority vs Router output on the same golden set ≥ 85%
Escalation appropriateness Security / ambiguous tickets must land Hand 3; high-similarity repeats must not be forced to Hand 3 100% on policy cases
Resolution usefulness Requester feedback (Worked / Did not work) + assignee outcome; proxy: repeat Hand 1 rate drops for same pattern ≥ 70% positive on Hand 1 demos
End-to-end latency Ticket submit → Hand assigned + UI updated; audit timestamps per agent step < 15 s typical (demo env)

The low-level design defines audit fields and reporting hooks; Round 3 adds live dashboards and expanded golden sets.

Scope — future features (out of core delivery)

Outside the current product backlog; extensions we plan when capacity allows.

  • Richer sentiment and adaptive thresholds from feedback volume.
  • Ticket clustering / trend alerts for operations.
  • Controlled auto-remediation where policy explicitly allows.
  • Multi-region DB, SSO, and full SOC-style logging.

Design principles

  • Human-centered — the right Hand, immediately.
  • Minimal — Three Hands, five AI agents, one internal score driving decisions.
  • Honest — escalation when unsure; confidence on detail, not noise on the home screen.
  • Coherent — Hands name outcomes; low-level design carries DDL, emails, APIs.
