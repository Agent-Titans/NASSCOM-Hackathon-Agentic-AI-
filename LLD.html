Low-level design

Five AI agents, Three Hands, Gemini RAG, SQLite + ChromaDB — the full technical blueprint for ticket routing and resolution.

Differentiator: governance-first agentic design — five specialized AI agents, Supervisor-controlled confidence routing into Three Hands, RAG-grounded resolution, and explicit guardrail + audit controls for responsible AI.

Companion docs: Architecture flow.html · Open source tools and libraries.html

This LLD is a diagram-first, end-to-end technical design for the application architecture: Streamlit UI plus an in-process AI agent pipeline (Guardrail -> Classifier -> Router -> Resolver -> Supervisor), with Gemini RAG generation and ChromaDB retrieval. Transactional state lives in SQLite (SQLAlchemy); vectors and similar-ticket search live in ChromaDB. Notifications execute the chosen Hand.

Design note: Diagrams are the star. Text is dense, concise, and method-signature focused.

This is a Round 2 blueprint for evaluation; production hardening (high concurrency, distributed services, and enterprise observability) is a Round 3 track.

Supervisor decision matrix

The decision matrix is based on weighted confidence c_total.

c_totalPathEngineNotes
>= 0.80Hand 1Resolver AI agent + notification serviceHigh similarity, direct self-help message with embedded success/failure track buttons
0.60 - 0.79Hand 2Router AI agent + department queuesRoute to department with suggested steps and SLA
< 0.60Hand 3Low confidence or policy triggers human review

Path definitions

  • Hand 1: Resolver AI agent returns candidate resolution steps from Gemini RAG, and the notification service sends the first self-help message.
  • Hand 2: Router AI agent assigns the ticket to the correct department queue (with SLA) and the UI displays suggested steps and references.
  • Hand 3: Supervisor AI agent escalates to a human queue and the notification service triggers an escalation notice.
def supervise(self, similarity: float, sentiment: float, historical_success: float) -> float:
    return (similarity * 0.6) + (sentiment * 0.2) + (historical_success * 0.2)

UI/UX Layer

The UI is the surface for the chosen Hand outcome; it does not expose internal agent math.

Requester flow (role-based, Streamlit)

  1. Home: My tickets list shows status and selected Hand (1/2/3).
  2. New ticket: create ticket with description + urgency.
  3. Ticket detail:
  • human-readable confidence line derived from internal c_total
  • Hand 1: ordered self-help steps + feedback controls
  • Hand 2: routing notice + expected response time + references
  • Hand 3: escalation status + audit context summary

Assignee flow (IT/support teams)

  1. Queue: department queue filtered by priority and SLA remaining.
  2. Ticket detail:
  • grounded suggestions when available
  • similar past tickets for fast triage
  • escalation flags + SLA clock

3 Hands integration

The chosen Hand is always produced by Supervisor AI agent after the five-step pipeline finishes.

Hand 1: Self-help (high trust)

  • Guardrail AI agent sanitizes for safe prompts/logs
  • Classifier AI agent identifies the technical domain
  • Resolver AI agent (Gemini RAG) generates ordered steps + citations
  • Supervisor AI agent selects Hand 1 based on c_total band
  • Persistence stores the resolution artifacts for the UI
  • Notification service executes Hand 1 messaging (including success/failure track buttons)

Hand 2: Routed assist (medium trust)

  • Same pipeline order as Hand 1
  • Router AI agent assigns the department queue + SLA
  • Supervisor AI agent selects Hand 2
  • Persistence attaches suggested steps + reference tickets for assignees
  • Notification service executes Hand 2 routing messaging and SLA expectations

Hand 3: Human escalation (low trust or policy)

  • Triggered when retrieval grounding is weak or policy requires a person
  • Requester sees a straightforward escalation status (no forced auto-steps)
  • Assignee sees audit context and any assistive artifacts marked as "assist"

AI agent integration

Agent Pipeline

Fixed order: Guardrail AI agent -> Classifier AI agent -> Router AI agent -> Resolver AI agent -> Supervisor AI agent

Intermediate artifacts:

  • SanitizedText
  • ClassificationResult (use_case_category, subcategory, confidence_hint)
  • RoutingResult (department_queue, priority, sla_hours)
  • ResolutionResult (ordered steps, citations, low_grounding)
  • SupervisorDecision (Hand, c_total_band, escalation_required, UI status)

Orchestrator boundary (single entry point):

Guardrail AI agent

Purpose: sanitize raw ticket text so downstream embedding, retrieval, and Gemini prompts never receive unsafe content.

Implementation (core rules):

  1. PII scrubbing: emails/phone numbers/ID-like strings -> redacted placeholders
  2. Secrets scrubbing: token/key/password-like patterns -> redacted placeholders
  3. Prompt-injection hardening: treat instruction-like phrasing inside ticket text as data

Error handling: If sanitization cannot guarantee safety, the system routes to Hand 3 (human queue).

4.4.1 Classifier Agent

Purpose: Categorize a ticket into predefined technical categories (and subtypes).

Implementation: Primary: Google Gemini returns strict JSON (use_case_category, subcategory, confidence_hint).

Categories (from ticket language):

  • Network issue
  • Database issue
  • Data issue
  • Server issue
  • Hardware issue
  • Software issue
  • Access Management issue
  • Security issue

Mapping to app 7 domains:

  • Network -> Network
  • Server/Hardware -> Infrastructure
  • Software -> Application
  • Database -> Database
  • Data -> Storage
  • Security/Access -> Security / Access Management

Fallback (always available):

  • regex tokenization
  • keyword sets per category/subcategory
  • TF-IDF-like scoring (either with sklearn or a lightweight custom scorer)
  • default behavior: pick the first category only when no matches exist

API parameters (example, Gemini):

  • max_output_tokens=100
  • temperature=0.5

Error handling: If the Gemini call fails or JSON schema parsing fails, immediately run fallback keyword matching (no second LLM provider).

4.4.2 Router Agent

Purpose: Assign to a department queue (and SLA) using classification + urgency.

Implementation: Deterministic rules:

  1. use_case_category -> department queue list
  2. urgency mapping -> priority (P0/P1/P2) -> sla_hours
  3. policy overrides (Security and Access Management prioritize specialists)

Error handling: If a category has no routing rule, Supervisor AI agent selects Hand 3.

4.4.3 Resolver Agent

Purpose: Generate suggested resolution steps grounded in past tickets and KB chunks.

Implementation (Gemini RAG):

  1. Create embedding from SanitizedText
  2. Query ChromaDB for top_k similar chunks (filter by category when available)
  3. Build one prompt: ticket + retrieved snippets + configured policy text
  4. Call Gemini to generate ordered steps + citations/references
  5. Output ResolutionResult with low_grounding flag if retrieval is weak

Error handling:

  • Gemini timeouts -> mark low_grounding=true
  • Empty/weak retrieval -> mark low_grounding=true (Supervisor avoids confident auto-close)

4.4.4 Supervisor Agent

Purpose: Quality validation and final Hand/status selection.

Decision logic: c_total = (similarity * 0.6) + (sentiment * 0.2) + (historical_success * 0.2)

Bands:

  • c_total >= 0.80 -> Hand 1
  • 0.60 <= c_total < 0.79 -> Hand 2
  • c_total < 0.60 -> Hand 3

Additionally:

  • if low_grounding=true OR policy requires human review -> Hand 3

Persist:

  • selected Hand
  • c_total_band
  • escalation_required

Database Schema

Canonical workflow state lives in SQLite; agent outputs persist in normalized tables.

Example SQLAlchemy-style models (field-level design):

class Ticket(Base):
    __tablename__ = "tickets"
    ticket_id = Column(String, primary_key=True)
    requester_id = Column(String, index=True, nullable=False)
    category = Column(String, nullable=False)
    subcategory = Column(String, nullable=True)
    urgency = Column(String, nullable=False)  # e.g. low/medium/high
    priority = Column(String, nullable=True)  # P0/P1/P2
    sla_hours = Column(Integer, nullable=True)
    status = Column(String, nullable=False)   # RECEIVED/RESOLVED/ESCALATED

    created_at = Column(DateTime, nullable=False)

class AgentRun(Base):
    __tablename__ = "agent_runs"
    run_id = Column(String, primary_key=True)
    ticket_id = Column(String, index=True, nullable=False)
    guardrail_ok = Column(Boolean, nullable=False, default=False)
    classification_ok = Column(Boolean, nullable=False, default=False)
    routing_ok = Column(Boolean, nullable=False, default=False)
    resolver_ok = Column(Boolean, nullable=False, default=False)
    supervisor_ok = Column(Boolean, nullable=False, default=False)
    created_at = Column(DateTime, nullable=False)

class ResolutionArtifact(Base):

    __tablename__ = "audit_log"

Transactions

Transaction boundaries focus on keeping DB locks short:

  1. Ticket create transaction (short)

- insert Ticket - insert AgentRun row with defaults - commit

  1. AI pipeline execution (no long locks)

- run 5 AI agents in memory

  1. Artifact persist transactions (short per artifact group)

- persist classification, routing, resolution steps, audit events

  1. Final status transaction (short)

- update Ticket.status, Ticket.hand, confidence_band - commit

Idempotency:

  • each AgentRun is keyed by run_id and ticket_id
  • retries on AI timeouts never duplicate final status without matching run_id

Error handling

Unified error strategy:

  • Agent-level errors are converted into safe fallback outcomes
  • Supervisor chooses the safe Hand when critical artifacts are missing

Typical cases:

  1. Classifier output invalid JSON -> fallback keyword classifier -> set confidence_hint=low if still uncertain
  2. Chroma retrieval empty -> Resolver marks low_grounding=true
  3. Gemini timeout -> Resolver marks low_grounding=true
  4. Guardrail safety failure -> route to Hand 3 immediately

Performance

Time-saving mechanisms (what reduces end-to-end delay):

  • embeddings + retrieval done once in Resolver, not repeatedly
  • caching embeddings for identical/near-identical sanitized tickets
  • top_k kept small (e.g., 8) to reduce Gemini prompt size
  • async notifications after final Hand selection

Practical performance targets:

  • classify+route: < 1.5s typical
  • retrieval: < 1.0s typical
  • Gemini generation: bounded with timeouts and max_tokens
  • overall: show a "processing" UI state while pipeline runs

Security

Implementation-level security controls:

  1. PII safety first: Guardrail is the only input allowed into embedding and Gemini prompts
  2. Prompt injection hardening: ticket content is treated as data, not instructions
  3. Auditability: every critical decision stored as append-only audit events
  4. Role-based access: requester sees only their tickets; assignee sees department queues only

Authentication scope: Round 2 covers baseline role-based authentication and authorization flows for evaluation. Enterprise SSO, MFA, and hardened session lifecycle controls are a Round 3 hardening track.
